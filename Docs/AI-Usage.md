# Использование AI в разработке

## 1. Какие AI-агенты использовались

В рамках выполнения тестового использовался один AI-агент:

- `Codex / GPT-5 based coding agent`  
  Использовался как инженерный помощник для проектирования архитектуры, генерации каркаса модуля, реализации runtime-кода, устранения ошибок интеграции и подготовки документации.

Отдельные специализированные агенты или внешние LLM-сервисы дополнительно не подключались.

## 2. Краткое описание workflow

Работа с AI была построена поэтапно, а не в формате "сгенерировать весь проект сразу".

Использованный workflow:

1. Сначала через AI была зафиксирована архитектура модуля:
   - публичный API;
   - слои `Api / Domain / Application / Presentation / Infrastructure / Installers`;
   - границы ответственности между UI, игровой логикой и внешним вызовом.
2. Затем AI использовался для генерации чистого каркаса:
   - структура папок;
   - основные классы и интерфейсы;
   - Zenject installers;
   - `ScriptableObject` конфиги;
   - фасад внешнего запуска.
3. После этого AI по слоям реализовывал конкретные части:
   - доменную логику матча;
   - reward flow;
   - UI runtime builder;
   - demo host;
   - Addressables integration.
4. На каждом этапе результат не принимался "как есть":
   - код компилировался;
   - runtime поведение проверялось в Unity;
   - при проблемах формулировался следующий узкий prompt на исправление.
5. Финальная доводка шла через цикл:
   `ошибка / визуальный дефект -> точечный prompt -> правка -> повторная сборка -> повторная проверка`.

Что именно делегировалось AI:

- проектирование модульного API;
- генерация шаблонного и инфраструктурного кода;
- wiring через Zenject;
- реализация игровой логики и сессионного orchestration;
- создание runtime UI и demo-сценария;
- диагностика compile/runtime/layout ошибок;
- подготовка README и UML.

## 3. Примеры промптов

Ниже примеры реальных типов запросов, которые использовались в работе.

### Пример 1. Архитектура и каркас

```text
На основе выбранной архитектуры:
1. создай структуру папок;
2. создай все основные файлы-заготовки;
3. создай интерфейсы, модели данных, базовые классы и каркас модульного API;
4. подготовь точки расширения под UI, игровую логику, награды и внешний запуск.
```

### Пример 2. Реализация игровой логики

```text
Реализуй доменную и прикладную логику миниигры:
- поле 3x3;
- очередность ходов;
- валидацию ходов;
- определение победы / поражения / ничьей;
- защиту от некорректных кликов;
- завершение матча;
- простого AI-соперника.
```

### Пример 3. Внешний API и интеграция

```text
Реализуй главный сценарий: запуск миниигры из внешнего кода.
Нужно:
- сделать публичный API модуля;
- внешний код должен иметь возможность вызвать миниигру не только со старта сцены;
- внешний код должен получить результат завершения;
- API должен быть чистым, простым и не зависеть от внутренних деталей UI.
```

### Пример 4. Точечная правка ошибки интеграции

```text
исправь ошибку

```

### Пример 5. Точечная правка UI/layout

```text
Исправь только layout и presentation UI у demo-сцены и встроенного модуля TicTacToe.
Игровую логику не меняй.
...
карточка миниигры должна уменьшаться пропорционально с сохранением aspect ratio, а не обрезаться.
```

## 4. Как проверялись результаты генерации

Проверка была многоступенчатой.

### Компиляция

После значимых изменений выполнялась сборка C#-проектов:

- `TicTacToeModule.Runtime.csproj`
- `Assembly-CSharp-Editor.csproj`

Это использовалось для раннего отлова:

- ошибок сигнатур;
- пропущенных файлов;
- проблем с namespace;
- ошибок DI wiring;
- проблем после рефакторинга runtime/editor-кода.

### Ручная проверка в Unity

Проверялось следующее:

- открывается ли `SampleScene`;
- корректно ли работает `SceneContext`;
- создается ли demo host;
- встраивается ли миниигра именно в `Module Host`;
- возвращается ли наружу `TicTacToeResult`;
- отображается ли reward;
- корректно ли работает повторный запуск.

### Проверка интеграции Addressables

Отдельно проверялось:

- создается ли `TicTacToeScreen.prefab`;
- назначается ли `AssetReference` в `TicTacToeModuleSettings.asset`;
- собирается ли addressable content;
- освобождаются ли preload/runtime handles.

### Проверка UI по скриншотам

На финальной стадии часть layout-ошибок исправлялась по фактическим скриншотам запущенной сцены:

- выход миниигры за пределы host-контейнера;
- modal-оверлей вместо embedded-widget;
- клиппинг карточки;
- неправильные якоря;
- схлопывание высоты секций;
- слишком рыхлая левая колонка demo.

## 5. Какие ответы агента пришлось корректировать

Наиболее заметные случаи, где сгенерированное решение пришлось дорабатывать:

- `Zenject binding issue`  
  AI сначала зарегистрировал `TicTacToeModuleService` через несколько `AsSingle`, что привело к ошибке `Zenject 6+`.  
  Исправление: singleton был оставлен только на concrete binding, интерфейсы переведены на `FromResolve()`.

- `устаревшие built-in ресурсы Unity`  
  В runtime UI использовались `Arial.ttf` и `UISprite.psd`.  
  В используемой версии Unity это привело к runtime-ошибкам.  
  Исправление: замена на `LegacyRuntime.ttf` и отказ от зависимости на built-in sprite.

- `ошибка embedded UI`  
  Первая версия UI вела себя как modal-окно, а не как встроенный модуль.  
  Исправление: root был нормализован под host-контейнер, убран standalone `Canvas`, добавлен clipping и пересобран layout.

- `layout conflicts`  
  В сгенерированных builder’ах смешивались `LayoutGroup` и ручной `Stretch`, из-за чего текст, кнопки и поле накладывались друг на друга.  
  Исправление: layout был полностью пересобран с корректным использованием `LayoutElement`, `VerticalLayoutGroup`, `HorizontalLayoutGroup` и централизованного fit-to-parent поведения.

- `проблема со старыми prefab/layout версиями`  
  После правок layout старые auto-generated hierarchy продолжали использоваться.  
  Исправление: добавлен `layoutVersion`, а addressable prefab стал пересобираться через editor utility.

- `новый runtime-файл не попал в Unity-generated csproj`  
  При добавлении вспомогательного UI-компонента сборка не находила класс, потому что Unity еще не обновила `.csproj`.  
  Исправление: вспомогательный компонент был встроен в уже существующий runtime-файл, чтобы не зависеть от времени перегенерации project files.

## 6. Как решение доводилось до финального качества

Доводка шла не через один "идеальный" prompt, а через серию инженерных итераций.

### Этап 1. Стабилизировать архитектуру

Сначала был зафиксирован контракт:

- внешний код работает только через `ITicTacToeLauncher` / `ITicTacToeModule`;
- игровая логика не зависит от UI;
- UI не знает про внешнюю интеграцию;
- конфиги и runtime wiring идут через Zenject.

Это позволило потом безболезненно править UI и инфраструктуру, не ломая игровую часть.

### Этап 2. Добиться надежной сборки

После каждой заметной серии правок проверялась компиляция.  
Все проблемы уровня "не собирается" устранялись до перехода к следующему шагу.

### Этап 3. Довести runtime-интеграцию

Особое внимание уделялось не генерации классов как таковой, а поведению модуля в host-приложении:

- preload;
- launch;
- cancel/close;
- release;
- повторный запуск;
- возврат результата;
- работа внутри контейнера UI.

### Этап 4. Довести presentation до приемлемого demo-качества

UI не принимался по факту "он отрисовался".  
Он доводился до состояния, пригодного для показа:

- две отдельные demo-колонки;
- embedded widget вместо modal overlay;
- квадратное поле `3x3`;
- корректное размещение кнопок и статусов;
- читабельная host-панель для записи видео.

### Этап 5. Зафиксировать решение в документации

После стабилизации кода AI использовался уже как инструмент упаковки результата:

- `README.md`;
- UML;
- описание процесса использования AI.

Итоговый подход был таким: AI использовался не как "генератор готового проекта", а как ускоритель инженерного цикла с обязательной ручной валидацией архитектуры, компиляции и runtime-поведения.
